Capitulo 1: Motivación

Control de versiones. Definición y objetivos.

Sin control de versiones.

  • Copiar ficheros y mandar correos no escala.
  • Sólo tenemos la versión actual.
  • Colaborar es dificil. ¿Quién tiene la última versión?

Control de versiones. Definición y objetivos.

Con control de versiones

  • Se mantiene un historial de cambios y colaboraciones.
  • Se puede colaborar de manera ordenada con otros. Manteniendo un registro de cambios.
  • Permite realizar metricas acerca de como crece un proyecto y colabora un equipo.
  • Ayuda a mejorar la gestión de cambios y los aspectos de configuración del software.
  • Ayuda a mejorar la calidad del software.

Historia de Git

  • Git surge en el entorno del desarrollo de Linux.
  • antes de 02/2002. Para desarrollar Linux, se usaban parches y tar.gz (y sin UML ni tonterias de esas.)
  • 02/2002.Bitmover regala la licencia de BitKeeper un SCM distribuido privativo.
  • 04/2005. BitMover retira la licencia tras roces.
  • 04/2005. Linus Torvalds presenta Git, que ya reúne ramas.
  • 06/2005. Git se usa para gestionar Linux
  • 02/2007. Git 1.5.0 es utilizable por mortales
  • 11/2011. Git 1.7.7.3

Ventajas de un SCV distribuido

  • Rapidez. Todo se hace en local, el disco duro es más rapido que la red.
  • Entorno de trabajo independiente. Nadie ve nada nuestro hasta que lo mandamos, sin embargo . . .
  • Podemos ir haciendo revisiones pequeñas intermedias.
  • Podemos hacer experimentos sin afectar al resto del equipo o al resto del proyecto.

Ventajas de un SCV distribuido II

  • Trabajo sin conexión En el tren, avión, en casa de tus suegros, etc.
  • Robustez hay tantas copias como repositorios nos hayan clonado.

GIT: Primeros pasos

Crear un repositorio

	    $ mkdir DIR
	    $ cd DIR
	    DIR $ git init
	    > Initialized empty Git repository in DIR

Creamos algo

	    $ touch fichero.txt
	    $ echo "Winter is coming" >> fichero.txt

Primero hay que añadirlo al area de preparacion

	    $ git add fichero.txt

Capitulo 3: lo interno

Modelo de datos de Git

  • Un repositorio es un grafo orientado acíclico de objetos.
  • Hay 4 tipos de objetos: commit, tree, blob y tag
  • Los objetos son direccionables por contenido (resumen SHA1)
  • Los objetos son inmutables: al cambiar su contenido, cambia su SHA1
  • Git no gestiona información de ficheros movidos.
  • Git almacena los objetos de manera eficiente. Nunca guarda más de un objeto, aunque aparezca en muchos sitios

Capitulo 4: Ramas, merge y rebase.

Concepto: ramas.

  • Crear y utilizar ramas es parte del modelo de datos, no un apaño con directorios.
  • Rama en Git: referencia mutable y compartible a una revisión
  • Etiqueta en Git: referencia inmutable a un objeto
  • Cuando creamos un repositorio automaticamente se crea la rama master que es la principal.
  • Podemos crear ramas que tendran largo recorrido ( por ejemplo para mantener versiones antiguas )
  • O crear una rama para desarrollar una funcionalidad para luego desecharla

Reuniendo ramas con git merge: conflicto

  • Si dureante la reunión es imposible fusionar las ramas ocurre un conflicto.
  • Un conflicto ocurre cuando dos revisiones afectan a la misma zona de un fichero.
  • git mergetool lanza herramienta gráfica para resolver todos los conflictos.
  • Si el archivo es binario o si lo preferimos podemos utilizar:
    • git checkout --ours: nos quedamos con lo nuestro
    • git checkout --theirs: desechamos lo nuestro
    • Editamos el fichero. Dentro aparecen unos marcadores en la zona de conflicto indicando que contenia cada revisión
  • Preparamos con git add y subimos los cambios con git commit
